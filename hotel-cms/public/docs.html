hotelCMS

AI-Powered Hotel Website Builder

Create stunning hotel websites using pre-built components and AI that understands your brand. Describe what you want in plain language — hotelCMS handles the rest. No coding required.

12 Pre-built Components
4-Tier AI Routing
RFC 6902 Versioning
Visual Page Builder

hotelCMS is a full-stack platform that combines a visual drag-and-drop page builder with a multi-model AI engine. Hotel teams can manage their website through a CMS dashboard, while AI interprets natural-language commands to modify content, swap components, adjust themes, and even inject custom code — all version-controlled with instant rollback.

⚡
Features Overview

Everything you need to build, manage, and evolve a premium hotel website.

🧠
Multi-LLM AI Engine
4-tier task routing selects the cheapest model that can handle each task. Tier 0 hits the DB directly, Tiers 1–3 escalate to increasingly capable LLMs. Every action is classified before execution.
◧
12 Pre-built Components
Hero (cinematic, editorial, minimal), Rooms (grid, showcase), Gallery (masonry, filmstrip), Booking (inline, sticky), Reviews wall, Map, and Footer — all production-ready.
🖱
Visual Page Builder
Drag sections to reorder, toggle visibility, preview live in an iframe, and fire AI commands from anywhere via the ⌘K command bar.
✏
AI Direct Code Editing
AI goes beyond structured CMS operations — it can inject custom CSS targeting any element or replace a section's HTML entirely for pixel-perfect custom layouts.
💾
Hotel Context & Memory
AI remembers your brand voice, visual style preferences, and past decisions. Each interaction contributes to a growing hotel context that makes AI responses more accurate over time.
🕓
Version Control
Every change is stored as an RFC 6902 JSON patch. View the full version timeline, compare diffs, publish, reject, or rollback to any previous state with one click.
⚔
Smart Merge
Detects and handles conflicts when structured CMS edits collide with AI custom HTML/CSS overrides, preventing silent data loss.
🔗
Reference URL Generation
Paste a competitor or inspiration URL — the AI scrapes its structure and generates a full homepage with matching aesthetic and layout, adapted to your hotel's content.
🖼
Image Pipeline
Upload → EXIF strip → WebP + AVIF conversion → responsive srcset generation → AI-generated alt text and semantic tags. Images are optimized for Core Web Vitals out of the box.
🎨
Theme Editor
Color tokens, typography scales, spacing rhythm, and border radius — all editable with a live-preview swatch. Changes propagate instantly across all components.

Tech Foundation

Built on Next.js 16 with App Router and Server Components for zero-JS pages by default. PostgreSQL with Prisma ORM for typed database access. OpenRouter as the LLM gateway to swap between providers. Tailwind CSS for component styling.

Tier Routing Logic

Tier Trigger Handler Latency
Tier 0 Rule-matched: simple text/prop updates Direct DB write, no LLM call < 50ms
Tier 1 Structured CMS ops, low complexity Fast, cheap LLM (e.g. Haiku) ~400ms
Tier 2 Multi-step ops, theme changes, memory Mid-tier LLM (e.g. Sonnet) ~1.5s
Tier 3 Code injection, HTML replacement, complex reasoning Frontier LLM (e.g. Opus / GPT-4o) ~4s

Getting Started

Get hotelCMS running locally in under 5 minutes.

1
Install PostgreSQL
hotelCMS requires PostgreSQL 14+. Install via Homebrew (macOS), apt (Ubuntu), or use a managed service like Neon or Supabase.
# macOS
brew install postgresql@16 && brew services start postgresql@16

# Ubuntu
sudo apt install -y postgresql postgresql-contrib
2
Clone the repository and install dependencies
git clone https://github.com/your-org/hotel-cms.git
cd hotel-cms
npm install
3
Configure environment variables
Copy the example env file and fill in your credentials.
cp .env.example .env
# .env
DATABASE_URL="postgresql://user:password@localhost:5432/hotelcms"
OPENROUTER_API_KEY="sk-or-..."
NEXTAUTH_SECRET="your-secret-here"
NEXT_PUBLIC_APP_URL="http://localhost:3000"
4
Run database migrations
npx prisma migrate dev --name init
5
Seed demo data
Populate the database with a sample hotel, pages, and components.
npx prisma db seed
6
Start the development server
npm run dev
7
Open hotelCMS
Navigate to the CMS dashboard in your browser.
http://localhost:3000
💡
Press ⌘K anywhere in the dashboard to open the AI command bar. Try: "Make the hero section cinematic with a full-bleed image"

CMS Operations Reference

The AI emits one or more of these 13 typed operations in response to every user command. Operations are validated with Zod and executed atomically.

Operation Description Min Tier Key Payload Fields
update_text Change visible text content in a section (headline, body, CTA label) Tier 0 sectionId, field, value
update_props Update structured props (layout variant, colors, image URLs, boolean flags) Tier 0 sectionId, props (partial)
swap_component Replace a section's component type while preserving compatible props Tier 1 sectionId, newComponentId
add_section Insert a new section at a specified position on the page Tier 1 pageId, componentId, position, props
remove_section Delete a section from a page (soft delete, recoverable via rollback) Tier 1 sectionId
update_theme Modify global theme tokens: colors, typography, spacing, radius Tier 2 hotelId, themeChanges (partial Theme)
update_meta Update page SEO metadata: title, description, OG image, canonical Tier 0 pageId, meta
reorder_sections Resequence sections on a page by providing a new ordered array of IDs Tier 1 pageId, sectionIds (ordered)
inject_css Append scoped custom CSS targeting a specific section or global scope Tier 3 sectionId?, css
replace_html Fully replace a section's rendered HTML with AI-generated markup Tier 3 sectionId, html
clear_overrides Remove all custom CSS / HTML overrides for a section, reverting to component defaults Tier 1 sectionId
save_memory Persist a fact or preference to the hotel's AI memory store for future sessions Tier 2 hotelId, key, value
update_brand_voice Update the hotel's brand voice descriptor used in all subsequent AI content generation Tier 2 hotelId, brandVoice
ℹ
A single AI response can contain multiple operations that execute in order. For example, swapping a component and then updating its props happens in one atomic transaction.

Database Schema

9 Prisma models covering the full domain from hotel configuration to AI memory.

Model Purpose Key Fields
Hotel Root entity. One record per property. id, name, slug, domain, themeId, brandVoice
Theme Global design tokens for a hotel. primaryColor, fontFamily, borderRadius, spacing, colorTokens (JSON)
Page A published or draft page on the hotel site. slug, title, status, metaTitle, metaDescription, hotelId
Section A placed component instance on a page. pageId, componentId, order, props (JSON), customCss, customHtml, visible
Component Registry of all available UI components. name, slug, category, defaultProps (JSON), propSchema (JSON), minTier
Room Hotel room or suite record. name, description, pricePerNight, capacity, amenities, mediaIds
MediaAsset Uploaded images and videos post-processing pipeline. originalUrl, webpUrl, avifUrl, srcset, altText, tags, width, height
SchemaVersion Immutable version snapshot with RFC 6902 patch. patch (JSON), status (draft/published/rejected), createdBy, publishedAt
HotelContext AI memory store. Key-value pairs for persistent hotel preferences. hotelId, key, value (JSON), updatedAt

AI Prompt Flow

Every natural-language command travels through a deterministic pipeline before anything in the database changes.

1 — User types a command
Via the ⌘K command bar, a text field in the dashboard, or the API directly. Example: "Make the booking section sticky and change the CTA to 'Reserve Now'"
2 — Intent classification
A lightweight rule engine checks for pattern matches (regex + keyword heuristics). On a match, Tier 0 executes directly. On ambiguity, the classifier calls a small LLM to resolve intent and assign a tier.
3 — Context assembly
The system gathers: hotel profile + brand voice, active theme tokens, all sections with props, HotelContext memory entries, and the current rendered HTML of the page. This forms the LLM's system prompt.
4 — LLM call via OpenRouter
The appropriate tier's model receives the assembled context and the user command. It responds with a structured JSON array of operations conforming to the CMS operations schema.
5 — Response parsing and validation
The LLM output is parsed and each operation is validated with Zod. Invalid operations are rejected with an error surfaced to the user. Valid operations proceed to execution.
6 — Execute against database
Operations run in a Prisma transaction. A before-state snapshot is captured, the changes applied, and an RFC 6902 patch computed between old and new state for the version record.
7 — Preview refresh
The dashboard's iframe preview reloads, reflecting all changes. The version timeline updates with the new draft entry, ready to publish or reject.

Context Window Budget

// Approximate token allocation per LLM call
systemPrompt     = hotel profile + theme tokens      // ~800 tokens
sectionContext   = all sections serialized to JSON   // ~2,400 tokens
aiMemory         = HotelContext key-value pairs       // ~400 tokens
renderedHtml     = truncated current page HTML        // ~1,200 tokens
userMessage      = the natural language command       // ~50–200 tokens
// Total: ~5,000 tokens (well within Tier 1 context windows)
